Chapter 0

The Problem We're Solving

When you ask Nexus "why does login fail?", it needs to find the right functions to give the LLM as context. Get this wrong — return irrelevant code — and the LLM either hallucinates or gives a generic answer. Get it right, and the LLM traces the exact execution path to the bug.

The challenge is that similar code ≠ related code. The function you're asking about might only contain the entry point — the real logic is one or two function calls away.

Query
"why does login fail?"
→
Naive finds
login()
auth_middleware()
validate_token() ✗
check_permissions() ✗
💡
The core insight: The bug in a login failure is almost never in the login function. It's in validate_token() or check_permissions() — functions that login calls. A good retrieval system needs to follow the call chain, not just find similar names.

What good retrieval needs

  • ◆ Semantic recall — find functions whose meaning matches the query, even if they don't share keywords
  • ◆ Keyword recall — catch exact function names the user mentions, which embeddings sometimes blur
  • ◆ Causal coverage — include the functions the query function calls and callers, not just the function itself
  • ◆ Noise suppression — don't fill the context window with unrelated utility code
  • ◆ Precision over recall — 10 perfectly relevant nodes beats 50 vaguely relevant ones; the LLM context window is limited
Chapter 1

How We Measure Quality

Every pipeline change is validated against the same benchmark: RAGAS — Retrieval Augmented Generation Assessment — run on a handcrafted golden Q&A set. Here's what each metric actually tells you.

🎯
Context Precision
The most important metric
Of all the code nodes we retrieved, what fraction were actually relevant to the question? High precision = no noise in the context window.
Plain English
If you ask about authentication and we return 10 nodes — 4 about auth, 6 about logging — context precision is 0.4. We wasted 60% of the LLM's attention.
💬
Answer Relevancy
Does the answer address the question?
Measures whether the generated answer actually answers the question asked. Low relevancy means the LLM wandered off-topic or gave a generic response.
Plain English
Ask "where is OAuth2 defined?" and get a paragraph about HTTP in general — low answer relevancy. Get a direct answer pointing to security/oauth2.py — high relevancy.
🔒
Faithfulness
No hallucination
Measures whether every claim in the answer is grounded in the retrieved context. Low faithfulness = the LLM is making things up beyond what it was given.
Plain English
If the LLM says "the token expires in 30 minutes" but nothing in the retrieved code mentions expiry — that's an unfaithful claim, even if it happens to be true.

The Golden Set — golden_qa_v2

All evaluations run against 30 hand-authored Q&A pairs on the FastAPI source repo (5,006 nodes, 29,579 edges). The questions span five categories designed to stress-test different retrieval patterns:

Symbol navigation Class inheritance Parameter validation Middleware & DI ASGI routing
⚙
Eval setup: Judge model — ollama qwen2.5:7b (local). Embeddings — nomic-embed-text (local). All 3 runs use identical config so scores are directly comparable. Each run takes ~1–2 hours on a MacBook.

Chapter 2 — Generation 0
G0

Naive Vector Search

⬤ The Baseline

The starting point. Embed the query as a vector, search for the closest code nodes by cosine similarity. Simple, fast, and surprisingly hard to beat on some metrics.

How it works

1
Embed the query
The user's question is converted to a high-dimensional vector by nomic-embed-text. Every code node was embedded the same way at index time and stored in sqlite-vec.
2
Cosine nearest-neighbour search
Find the top-15 code nodes whose vectors point in the most similar direction to the query vector. That's the entire pipeline. No graph, no keyword search, no reranking.

Where it falls short

✗
Semantics ≠ causality. "Why does login fail?" finds functions that are semantically about login. But the bug is in validate_token() — a function login calls. The embedding for "validate a JWT token" and "why does login fail" are not close in vector space. Cosine search misses it entirely.
✗
Exact names get blurred. If a user asks "where is OAuth2PasswordBearer defined?", embedding the exact class name still produces a vector that might match generic authentication concepts. FTS keyword search would find it exactly — but naive vector has no FTS.
💡
Why it's still the baseline: Naive vector sets the floor. Every subsequent pipeline improvement is measured against it. Context precision of 0.1585 means roughly 16 out of every 100 retrieved nodes were actually relevant — there is a lot of room to improve.
Chapter 3 — Generation 1
G1

Graph RAG v1

↓ Context Precision Regression

The first attempt to add graph-awareness. The core idea was right: follow CALLS edges to find causally related code. But the implementation had three flaws that made context precision worse than doing nothing.

What was added

1
FTS5 keyword search added
BM25 keyword search over function and class names. Finds exact matches that cosine similarity misses. Merged with semantic results using max(semantic, fts).
2
BFS expansion via ego_graph
From seed nodes, NetworkX's ego_graph() expands to all neighbours within hop_depth=1. This includes both CALLS and IMPORTS edges — every import in the file becomes a candidate.
3
Dual-score reranking
Non-seed BFS nodes all got a flat score of 0.3 — completely independent of how relevant their parent seed was. Sorted by this score, then MMR.

Why context precision dropped — the three culprits

  • 1
    IMPORTS edges caused cross-file pollution ego_graph() followed both CALLS and IMPORTS edges. When you query about Depends(), BFS expansion includes every module that imports fastapi.params — logging, typing, and dozens of unrelated utilities all become candidates, flooding the result set with off-topic nodes.
  • 2
    Flat 0.3 score made all neighbors equally irrelevant Every BFS-expanded node got a score of exactly 0.3, regardless of how relevant its parent seed was. A neighbor of a highly relevant seed and a neighbor of a barely-relevant seed competed on equal footing. The scoring was query-independent.
  • 3
    ego_graph over-expansion With multiple seeds and BFS at depth=1 across all seed neighbors, ego_graph could pull in 50+ candidate nodes. More candidates meant more noise, and the flat 0.3 score meant there was no signal to rank them by.
💡
The lesson: Faithfulness and answer relevancy improved because FTS seeds were better. But context precision cratered because of graph expansion noise. Adding graph context is the right direction — but the expansion strategy was wrong.
Chapter 4 — Dead End
☠

HyDE + Cross-Encoder

⬤ Eval-Only, Never Shipped

Between v1 and v2, we tested a more complex eval-only pipeline combining two ideas: Hypothetical Document Embedding (HyDE) for query expansion, and an early cross-encoder reranker. It was promising on paper but never made it to production.

What HyDE does

Instead of embedding the raw user query ("why does login fail?"), HyDE first asks an LLM to generate a hypothetical code snippet that would answer the question. Then it embeds that hypothetical snippet for vector search. The idea: hypothetical code lands closer to real code in the embedding space than a natural language question does.

✗
Why it didn't ship: HyDE requires an extra LLM call for every query — adding ~400–800ms of latency before retrieval even starts. For a dev tool where 200–600ms to first token is the target, that's not acceptable. The precision gain (+11% vs naive) didn't justify doubling the latency budget.
💡
What it taught us: The cross-encoder component of this pipeline showed real promise for precision. That idea was carried forward — but applied after v2's cleaner graph expansion rather than on top of v1's noisy candidates.

Chapter 5 — Generation 2
G2

Graph RAG v2 — The Redesign

↑ First pipeline to improve all three metrics

A ground-up redesign that fixed the three problems in v1. Every change was surgical — each one directly addressed a specific failure mode identified from v1's RAGAS regression.

Four targeted fixes

  • 1
    RRF replaced max() merge Cosine scores live in [0,1]. BM25 scores have a different scale entirely. The old max(semantic, fts) merge was comparing apples and oranges. Reciprocal Rank Fusion merges by rank position, not score value — immune to scale differences between the two retrieval systems.
  • 2
    CALLS-only expansion (IMPORTS removed) BFS now follows only type == "CALLS" edges. Removing IMPORTS edges eliminates the cross-file pollution that destroyed v1's context precision. Querying about Depends() no longer pulls in the entire import graph of every file that uses it.
  • 3
    Propagated scoring replaced the flat 0.3 fallback Each neighbor now inherits its score from its parent seed: neighbor_score = parent_rrf_score × 0.6. A neighbor of a highly-relevant seed scores ~0.54. A neighbor of a barely-relevant seed scores ~0.12. The graph expansion is now query-aware.
  • 4
    Per-seed neighbor cap (5 callers + 5 callees) Instead of ego_graph expanding all neighbors of all seeds indiscriminately, each seed gets at most 5 top callers + 5 top callees (ordered by PageRank). Prevents high-degree hub nodes from flooding the candidate pool with their entire neighborhood.

Chapter 6 — Previous Generation
G2.5

v2 + Cross-Encoder Reranking

↑ Largest single-step precision gain at the time

After v2 fixed graph expansion, the remaining bottleneck was the ranking signal itself. Cosine similarity encodes query and document separately — it can't see how a specific query and a specific node interact with each other. A cross-encoder fixes this.

The key idea

📝
Bi-encoder (cosine)
What we had
Query → vector. Document → vector. Compare vectors. Each is encoded independently — they never see each other during scoring.
Analogy
Two people read the same book separately and write summaries. Then you compare summaries. Fast, but nuanced interactions between query and document get lost in the summaries.
🔬
Cross-encoder
What we added
Query + document are read together in one pass. The model attends to how specific terms in the query relate to specific parts of the document.
Analogy
One expert reads both the question and the document simultaneously and scores how well they match. More accurate — but can't pre-compute document vectors, so only practical for reranking a small candidate set.

How it fits into the pipeline

The cross-encoder doesn't replace the retrieval pipeline — it runs after v2's candidate pool is assembled. It takes the top 2×max_nodes candidates and rescores them. MMR diversity selection then operates on the CE-scored list.

Pipeline Order
1. Semantic search + FTS5 → RRF merge seeds
2. CALLS expansion → propagated scoring candidates
3. Cross-encoder rerank (top 2×N pairs) ↑ new
4. MMR diversity → final N nodes result

Startup pre-warming

The model (ms-marco-MiniLM-L-6-v2, 66 MB) is loaded in a background daemon thread when the FastAPI server starts. By the time the first query arrives, the model is already in memory (~200ms disk load). If pre-warm fails, the model loads on first request — no crash, just that one request pays the cold-start cost.

Chapter 7 — Retrieval Hardening
G3

Graph RAG v3 — Three Retrieval Filters

↑ +22% context precision over v2+CE

v2+CE left three retrieval problems unsolved. Low-similarity embeddings were still entering the candidate pool as noise. CE scores below zero — meaning the cross-encoder judged the node negatively relevant — were still being passed to MMR. And functions truncated at ingestion time were being fed as incomplete context to the LLM. v3 addresses all three.

Improvement ① — Cosine Similarity Floor

Semantic search now discards any candidate whose cosine similarity to the query falls below a minimum threshold (default 0.15). Before this, every indexed node returned by the ANN search would enter the pipeline regardless of quality. Nodes with similarity near zero add noise without contributing signal — filtering them early reduces the size of the candidate pool and means every subsequent stage (RRF, CE, MMR) works with higher-quality inputs.

💡
Why 0.15? Cosine similarity of 0.15 is roughly the noise floor for sentence-transformer embeddings on code — two completely unrelated functions often score 0.1–0.13. A floor of 0.15 preserves all genuinely related nodes while cutting about 15–20% of the candidate pool.

Improvement ② — CE Score Floor

After cross-encoder reranking, any node with a CE score ≤ 0 is dropped before MMR runs. A negative CE score means the cross-encoder's joint read of (query, node) produced a score that maps below the relevance midpoint — the model is actively saying this pair is not a match. In v2+CE, these nodes were still passed to MMR and could end up in the final result if the pool was sparse. The v3 floor prevents the CE from handing the LLM context it explicitly judged as irrelevant.

v3 Pipeline — CE Floor Added
1. Semantic search → cosine floor (≥ 0.15 only) ↑ new
2. FTS5 keyword search → RRF merge
3. CALLS expansion → propagated scoring
4. Cross-encoder rerank
4b. CE score floor — drop nodes with CE ≤ 0 ↑ new
4c. Full body expansion — replace truncated previews ↑ new
5. MMR diversity → final N nodes

Improvement ③ — Full Body Expansion

At ingestion time, large functions are truncated to a body_preview (the first ~500 chars) to keep index size manageable. At query time, v3 fetches the complete full_body from the database for every selected node before building the context string. This means the LLM sees the complete function implementation — no invisible truncation boundary mid-function. The full_body_expanded stat in the retrieval summary tracks how many nodes needed expansion per query (typically 4–5 out of 15).

Chapter 8 — Current Production
★

v3.1 — CLASS_CONTAINS + Personalized PageRank

↑ +48% over v3, +80% over v2+CE on context precision

v3's BFS expansion still had a fundamental problem: it was limited to depth-1 CALLS edges and had no awareness of class structure. If the best seed node was a class, its methods stayed invisible. If a method was the seed, its sibling methods and parent class were unreachable. v3.1 replaces BFS with Personalized PageRank — and first adds the structural edges PPR needs to be useful.

New graph edges: CLASS_CONTAINS

Both the Python and TypeScript AST parsers now emit CLASS_CONTAINS edges from each class node to its method nodes, detected by line-range containment at parse time. graph_builder.py resolves these directly (both sides are already full node_ids) via a new _add_class_contains_edge() helper. This means the graph now has structural topology that PPR can traverse: a class and all its methods are a tightly connected cluster.

🌲
BFS Expansion (v3.0)
What we had
Depth-1 only. From each seed: follow CALLS edges one hop. Fixed 0.6× score decay. Could not reach class siblings, grandparent callers, or discover methods via their class.
Analogy
Ask your immediate manager and direct reports. No one else in the org, even if they're in the same team and highly relevant.
🔀
Personalized PageRank (v3.1)
What we built
Global random-walk seeded from RRF scores. Traverses CALLS + CLASS_CONTAINS. Multi-hop naturally. Returns top-30 non-seed neighbors by PPR score. Seeds teleport proportional to their RRF relevance.
Analogy
A random walker starts at your top search results, proportional to how relevant they are. They wander the code graph — following calls, stepping into classes and their methods — and the nodes they visit most often are returned as context.

How PPR works in the pipeline

PPR is seeded proportionally from the RRF-unified scores: the teleportation distribution is the normalized RRF score vector. The walk traverses only CALLS and CLASS_CONTAINS edges — IMPORTS are excluded (same reason as v1: cross-file pollution). The result is the top-30 non-seed nodes by PPR score. These are merged into the candidate pool (seed RRF scores take precedence on overlap) before the CE rerank.

v3.1 Pipeline — Full Picture
1. Semantic search → cosine floor (≥ 0.15)
2. FTS5 keyword search → RRF merge
3. PPR expansion — CALLS + CLASS_CONTAINS, top-30 neighbors ↑ replaces BFS
4. Cross-encoder rerank → CE floor (drop CE ≤ 0)
4b. Full body expansion — replace truncated previews
5. MMR diversity → final N nodes

Balanced grounding prompt

With PPR providing richer multi-hop context, the original strict grounding rule ("use ONLY the provided code context") became a liability — it forced the LLM to refuse to explain code patterns it could see but couldn't connect, producing canned refusals that RAGAS scored as low faithfulness. The revised prompt uses softer framing: prefer retrieved context as the primary source, allow general programming knowledge to explain patterns, never state codebase-specific facts not in context. The eval confirmed this: faithfulness jumped from 0.0455 (strict prompt) to 0.9133.

⚠
The strict-prompt trap: PPR surfaces callers-of-callers and class siblings — rich multi-hop context. The LLM can see, say, a base class and all its subclasses. A strict "ONLY cite retrieved context" rule meant it couldn't explain why the abstract method pattern exists, even though the pattern was visible. It would fall back to the canned refusal phrase, which RAGAS judged as faithfulness = 0. The balanced prompt unlocks the context PPR worked hard to find.

Reference

All 10 Techniques Explained

Every building block in the current pipeline — what it is, why it exists, and how to think about it without code.

📐
Cosine Similarity
Used in: all pipelines (Step 1)
Measures how similar two embeddings are by the angle between them. Score of 1.0 = identical direction. Score of 0 = perpendicular (unrelated). The primary retrieval signal in every pipeline.
Analogy
Every topic is a compass direction. "Authentication" and "login security" point in roughly the same direction. "Logging utilities" points somewhere else. Cosine similarity is the angle between two compass bearings — small angle = related topics.
🔤
BM25 / FTS5
Used in: v1, v2, v2+CE (Step 2)
Keyword search that counts how often your search terms appear in a document, with diminishing returns for repetition and a penalty for very common terms. Catches exact identifier names that embedding models blur.
Analogy
A librarian who counts how many times your exact terms appear. "OAuth2PasswordBearer" appearing once in a file title is a strong signal. "The" appearing 100 times means nothing. BM25 handles both cases correctly.
⚖️
Reciprocal Rank Fusion
Used in: v2, v2+CE (Step 3)
Merges multiple ranked result lists using rank position, not score values. A node ranked #2 in semantic search and #3 in FTS scores higher than one ranked #1 in only one list. Formula: Σ 1/(k + rank) where k=60 dampens very high rank advantages.
Analogy
Two scouts each rank candidates independently. Instead of averaging their scores (which use different scales), you combine their rank positions. Consistently ranked high by both scouts beats a single scout's top pick. Scale-invariant and robust.
🌐
CALLS Graph Expansion
Used in: v1, v2, v2+CE (Step 4)
Starting from semantic seed nodes, follow CALLS edges in the code call graph to include direct callers and callees. Depth-1 only — one hop from any seed. IMPORTS edges excluded to prevent pollution.
Analogy
You found the manager (seed). Now follow the org chart one level to find who reports to them (callees) and who they report to (callers). The bug is often in someone in that immediate team, not in the manager's own code.
📊
Propagated Scoring
Introduced in: v2 (replaces flat 0.3)
BFS-expanded neighbors inherit their relevance score from their parent seed: neighbor_score = parent_rrf_score × 0.6. Neighbors of highly-relevant seeds compete aggressively; neighbors of weak seeds stay subordinate.
Analogy
If the manager (seed) was highly relevant (score 0.9), their team members start with a credibility of 0.54. If the manager was barely relevant (score 0.2), their team starts at 0.12 — and won't crowd out better candidates.
🎲
Maximal Marginal Relevance
Used in: v1, v2, v2+CE (final step)
Selects the final N nodes iteratively — always picking the next candidate that is most relevant and least similar to already-selected nodes. Uses file_path as a diversity proxy: each additional node from the same file gets a 0.35 penalty per prior selection.
Analogy
You're assembling a panel of 10 experts. After choosing the top auth expert, the next person from the same team gets penalized — you'd rather add a routing expert than a second auth expert who covers the same ground.
🔬
Cross-Encoder Reranking
Introduced in: v2+CE (Step 4)
Runs the full text of each (query, node) pair through a transformer model simultaneously. Unlike cosine similarity which encodes them separately, the cross-encoder reads both together — catching query-document interactions that bi-encoders miss. Model: ms-marco-MiniLM-L-6-v2 (66 MB, ~100ms for 20 candidates on CPU).
Analogy
Bi-encoder: two people read the job description and résumé separately, then compare notes. Cross-encoder: one expert reads both together, line by line, noticing exactly where they align. The second approach catches subtle relevance signals the first approach misses in translation.
🚧
Cosine Similarity Floor
Introduced in: v3 (Step 1 filter)
Discards semantic search candidates whose cosine similarity to the query is below a minimum threshold (default 0.15). Applied immediately after the ANN search, before FTS or graph expansion. Reduces candidate pool size and ensures every downstream stage operates on candidates with at least minimal topical alignment.
Analogy
A job fair bouncer. Before any interview, they check if the candidate's résumé is even in the right industry. If it's completely unrelated, they're turned away at the door — saving the interviewers from reviewing nurse résumés for a software engineering role.
✂️
CE Score Floor
Introduced in: v3 (Step 4b filter)
After cross-encoder reranking, drops any candidate node with CE score ≤ 0. A negative CE score indicates the cross-encoder's joint (query, node) evaluation is below the relevance threshold — the model is explicitly saying this is not a match. Rather than let MMR inherit these negative-signal nodes, v3 removes them first.
Analogy
The expert interviewer who read both the job description and résumé together has scored this candidate negative. Even if the interview panel (MMR) needs to fill seats, this person is out. Don't let slot-filling override an explicit negative judgment.
🔀
Personalized PageRank (PPR)
Introduced in: v3.1 (replaces BFS)
A global random-walk over the code graph seeded proportionally from RRF-unified scores. Traverses CALLS and CLASS_CONTAINS edges (IMPORTS excluded). Returns top-30 non-seed neighbors by PPR score. Multi-hop by nature: discovers callers-of-callers, class siblings, and methods of retrieved classes — all weighted by their proximity to relevant seeds.
Analogy
Imagine a random walker who starts at your top search results, with a stronger pull toward the more relevant ones. They wander the code graph, following function calls and stepping into class hierarchies. The nodes they visit most frequently are likely the most structurally connected to your query — whether they're one hop or three hops away.
Deep Dive

FAQ — Understanding the Journey

The questions that unlock deeper understanding of each decision.

Why did Graph RAG v1 make context precision worse than doing nothing?
Three compounding problems hit at once. First, ego_graph() followed both CALLS and IMPORTS edges — so querying about Depends() expanded into every module that imports fastapi.params, flooding the pool with unrelated logging and typing utilities. Second, every BFS-expanded node got a flat score of 0.3, completely independent of whether its parent seed was relevant — so 50+ noisy neighbors all competed on equal footing. Third, there was no per-seed cap, so high-degree nodes (functions called everywhere) pulled in dozens of off-topic neighbors. The result: more nodes, worse signal. Context precision measures the fraction of retrieved nodes that were actually relevant — more noise directly lowers it.
Why does context precision matter more than faithfulness for our use case?
Faithfulness measures "did the LLM stick to what it was given?" — but if what it was given was mostly irrelevant, a high faithfulness score means it faithfully described irrelevant code. Context precision measures "was what we gave the LLM actually relevant?" — this is the retrieval quality signal that's independent of LLM behavior. For code navigation and debugging, the retrieval step is the hardest problem. Faithfulness can be improved by prompt engineering; context precision can only be improved by better retrieval.
Why does RRF work better than max() for merging semantic and FTS results?
Cosine similarity scores are in [0, 1]. BM25 scores (from FTS5) are in a completely different, unbounded range — the values are not comparable. max(semantic_score, fts_score) means the semantic score almost always wins because it happens to produce higher numbers, regardless of relevance. RRF is rank-based: it doesn't care about score values, only rank positions. A node ranked #1 by FTS and #4 by semantic gets a strong RRF score, even if its raw BM25 number is tiny. RRF is scale-invariant — it works correctly no matter how different the scoring ranges are.
Why CALLS-only? Couldn't IMPORTS edges also be useful?
In theory, yes — if you query about Depends, expanding to files that import it could find usage examples. In practice, IMPORTS edges create cross-file pollution at a massive scale. In the fastapi corpus, something like from typing import Optional creates an IMPORTS edge from every single file to typing.py. Following IMPORTS edges from any seed would include virtually the entire codebase as candidates. CALLS edges are much more selective — a function calls at most a handful of other functions, and those relationships are directly causally relevant to understanding the function's behavior. The precision gain from removing IMPORTS in v2 (+115% context precision vs v1) confirmed this decisively.
If we already have cosine similarity, why does the cross-encoder add so much?
Cosine similarity scores query and document independently — the model never sees how specific terms in the query interact with specific parts of the document. It's asking "are these two things generally about the same topic?" The cross-encoder reads them together and asks "does this specific document answer this specific question?" For code retrieval, the difference is significant: a function signature might be semantically related to authentication in general (high cosine score), but the cross-encoder can see that it doesn't actually implement the specific thing being asked about. The +46% context precision gain from CE directly reflects this — the CE is demoting high-cosine but low-relevance nodes.
Why does faithfulness barely move when we add the cross-encoder (+0.5%)?
Faithfulness measures whether the LLM sticks to what it was given — it doesn't care whether what it was given was good. Adding a cross-encoder changes which nodes are selected, but not how the LLM uses them. The LLM was already being faithful with v2's context; now it's faithful with a better context. The downstream effect shows up in answer relevancy (+17%) — better nodes → more precisely targeted answers — not in faithfulness, which was already near its ceiling for this model and prompt.
Why is the golden set only 30 questions? Is that enough?
30 is a pragmatic choice. Each question requires a full retrieval + LLM generation cycle with an Ollama judge — a full 30Q run takes ~1–2 hours locally. The questions were hand-authored to cover the five main query patterns (symbol navigation, inheritance, parameter validation, middleware/DI, routing) with ~6 questions per category. For the delta comparisons we care about (does this change help or hurt?), 30 well-chosen questions is sufficient to detect signal above noise. The consistent ordering — naive < v1 < HyDE < v2 < v2+CE on context precision — across a hand-labeled set gives high confidence the trend is real, not an artifact of small sample size.
What was the plan before v3 — and how did it change?
After v2+CE, the identified next steps were: (1) semantic filtering of CALLS neighbors — replace BFS's PageRank ordering with cosine similarity to the query; (2) actual source lines fetched at query time to eliminate truncation; (3) backward CALLS expansion. v3 delivered (2) as full body expansion. v3.1 replaced BFS entirely with PPR, making (1) partially redundant — PPR weights neighbors by seed-proximity already, not structural centrality. The remaining open work is query-aware CE filtering of expanded neighbors and chunk-level indexing for very large functions.
Why did PPR cause faithfulness to collapse before the prompt fix?
PPR surfaces multi-hop context — callers of callers, class siblings, inherited methods. The stricter "ONLY use the provided context" grounding prompt was written for v2's shallow BFS context, where a citation refusal was appropriate for anything not visible. With PPR's richer context, the LLM could often see the relevant pattern (e.g., a base class and all its subclasses) but couldn't explain the relationship without general programming knowledge. The strict rule forced a canned refusal phrase ("I'm not certain based on the retrieved context"), which the RAGAS faithfulness judge consistently scored as near-zero. The fix was to allow general programming knowledge to connect what's in context — not to invent codebase facts, but to explain patterns. Faithfulness went from 0.0455 → 0.9133.
What are CLASS_CONTAINS edges and why do they matter for PPR?
A CLASS_CONTAINS edge connects a class node to each of its method nodes. Before v3.1, the code graph only had CALLS and IMPORTS edges — so a class and its methods were structurally isolated unless they called each other. If a class was semantically similar to the query and became a seed, its methods were invisible to PPR. With CLASS_CONTAINS edges, the random walker naturally distributes mass across the class and all its methods, then follows CALLS edges from those methods outward — giving a complete neighborhood for class-centric queries.
What does the eval actually run — what's the judge doing?
The eval calls graph_rag_retrieve() (or the naive equivalent) for each of the 30 questions, generates an answer by passing the retrieved context to explore_stream(), then scores the (question, answer, retrieved_contexts, ground_truth) tuple using RAGAS. The RAGAS judge is ollama qwen2.5:7b running locally — it scores each sample on the three metrics by prompting the LLM to assess faithfulness, relevancy, and precision. Using a local Ollama judge means there's no API cost, but it also means the judge has its own biases and limitations. The scores are meaningful for relative comparisons between pipelines, not as absolute ground truth.
How do I run the eval after making a retrieval change?
Activate the eval venv: source venv_eval/bin/activate. For a quick sanity check (5 questions, ~15 min): python eval/run_ragas_v3.py --limit 5. For a full 30Q run (~2 hrs): python eval/run_ragas_v3.py --limit 30 --workers 1. To reuse a prior context precision result and only re-run faithfulness/relevancy (e.g., after a prompt-only change): python eval/run_ragas_v3.py --limit 30 --workers 1 --skip-context-precision. Results land in eval/results/ragas_v3_*.json. The comparison table printed at the end shows your run vs all prior baselines.
