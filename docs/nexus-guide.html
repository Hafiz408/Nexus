Explain Mode

Explorer Agent — Streaming explanation with citations

The Explorer streams tokens directly from the LLM, like a chat interface. Each answer includes clickable file:line citations. This is the only mode that bypasses LangGraph — it calls explore_stream() directly for minimum latency.

Example query→ Explorer Agent
"How does the auth middleware work?" → Graph RAG finds: auth.py::validate_token, auth.py::check_permissions, api.py::login_endpoint → Explorer streams explanation with citations to each function → You can click any citation to jump to that exact line in VS Code
Debug Mode

Debugger Agent — Ranked suspects with anomaly scores

The Debugger scores each retrieved function on 5 factors: complexity, missing error handling, keyword relevance, coupling, and low centrality. Returns top 5 suspects with a diagnosis narrative. Clicking a suspect opens that exact file and line.

DebugResult structure
class DebugResult(BaseModel): suspects: list[SuspectNode] # top 5, ranked by anomaly score diagnosis: str # narrative explanation traversal_path: list[str] # all BFS nodes visited (for Critic) class SuspectNode(BaseModel): node_id: str # "auth/login.py::validate_token" file_path: str line_number: int anomaly_score: float # 0.0 – 1.0 reason: str # why this function is suspect
Review Mode

Reviewer Agent — Structured findings with severity + fix suggestion

Uses with_structured_output(ReviewResult) to force the LLM to return valid JSON. Each finding has a severity (critical/high/medium/low), category (security/performance/style), and a concrete fix suggestion. You can post findings as inline PR comments via GitHub token.

ReviewResult structure
class ReviewResult(BaseModel): findings: list[Finding] class Finding(BaseModel): severity: Literal["critical", "high", "medium", "low"] category: Literal["security", "performance", "style", "correctness"] file_path: str line_number: int message: str suggestion: str # concrete fix (Critic requires this field non-empty)
Test Mode

Tester Agent — Writes test files directly to your repo

The Tester detects your test framework (pytest / jest / vitest), derives a logical test file path, generates test code, and writes it to disk using the MCP write_file tool. The Critic checks for at least one def test_ function before accepting.

Test file derivation logic
# Input: source file = "app/auth/login.py" # Output: test file = "tests/test_login.py" # Input: source file = "src/components/Button.tsx" # Output: test file = "src/components/Button.test.tsx" # Path traversal guard: checks raw string for ".." before Path.resolve() if ".." in test_file_path: raise ValueError("Path traversal detected")
Auto Mode

Router Agent — Three-path intent classification

The Router classifies intent via 3 paths: (1) user-supplied hint (no LLM call), (2) LLM classification with confidence ≥ 0.6, (3) low-confidence fallback → defaults to "explain". This makes explicit mode selection zero-cost.

Router classification paths
# Path 1 — User picked a mode in UI → instant, no LLM needed intent_hint = "debug" → IntentResult(intent="debug", confidence=1.0) # Path 2 — LLM classifies with high confidence "why does login fail?" → intent="debug", confidence=0.92 # Path 3 — Low confidence fallback to safest mode "tell me about this code" → intent="explain", confidence=0.41

Here is the complete journey of a single query from the moment you press Enter.

1

2

Search Engine

Graph RAG Retrieval

The core search innovation. Eight sequential steps in graph_rag.py turn a plain-English question into a precisely curated list of code nodes. The pipeline combines vector search, keyword search, rank fusion, graph random-walk, cross-encoder reranking, full-body expansion, and file-diversity selection.

The 8-Step Retrieval Pipeline

1
semantic_search — sqlite-vec cosine ANN
Query text is embedded with the configured provider → query vector serialized as float32 bytes → sqlite-vec virtual table returns top-k nearest neighbours by cosine distance.
Cosine floor: nodes with similarity < min_similarity=0.15 are dropped before entering the pipeline. Prevents completely unrelated nodes from polluting the seed set.
SQL (graph_rag.py)
-- sqlite-vec KNN query SELECT meta.node_id, vec.distance FROM code_embeddings_vec vec JOIN code_embeddings_meta meta ON meta.vec_rowid = vec.rowid WHERE meta.repo_path = ? AND vec.embedding MATCH ? AND k = ? -- score = 1.0 - distance (cosine); filter score < 0.15
2
fts_search — FTS5 BM25 keyword search
Query is tokenised into identifier-like words → stopwords removed (how, what, def, class, return, import, etc.) → FTS5 BM25 OR search over code_fts(name, embedding_text).
Results normalised to [0, 0.85] so perfect FTS matches yield below perfect semantic matches (1.0). Capped at 5 results — FTS is a complementary signal, not the primary one.
Why stopwords matter: "how does login work" → after filtering → "login", "work" → avoids flooding candidates with unrelated matches for "how" and "does".
3
rrf_merge — Reciprocal Rank Fusion (k=60)
The two ranked lists (semantic + FTS) are fused via RRF — a rank-based fusion immune to score scale differences between cosine [0,1] and BM25 scores. A node appearing high in both lists scores much higher than one topping only one.
RRF formula
rrf_score(node) = Σ 1.0 / (k + rank_i + 1) # k=60 empirically robust # rank_i = position in each list (0-indexed) # node in both lists: ~1/61 + ~1/61 = ~0.033 # node only in semantic top-1: ~1/61 = ~0.016
4
ppr_expand — Personalized PageRank graph expansion
Replaces the old depth-1 BFS from v2. Seeds RRF scores are used as teleportation weights in a global random-walk over CALLS + CLASS_CONTAINS edges (IMPORTS excluded to prevent cross-file pollution).
PPR advantages over BFS:
  • → Multi-hop: discovers callers-of-callers and class siblings naturally
  • → Score-proportional: neighbors of high-scoring seeds rank higher than neighbors of weak seeds
  • → CLASS_CONTAINS-aware: methods of retrieved classes (and vice versa) surface
Returns top-30 non-seed neighbors by PPR score (alpha=0.85).
PPR normalization
# Personalization = RRF scores normalized to sum to 1.0 personalization = {nid: s / total for nid, s in seeds.items()} ppr = nx.pagerank(subgraph, alpha=0.85, personalization=personalization)
5
Combine candidates + test-file penalty
Seeds (RRF scores) and PPR neighbors are merged into a single candidate pool — seeds overwrite neighbors on overlap (RRF scores are higher). Then a test-file penalty halves the score of any node whose file path contains "test" or "spec". This prevents test files from dominating context when the user asks about production code.
Test penalty
for node_id in candidate_pool: file_part = node_id.split("::")[0].lower() if "test" in file_part or "spec" in file_part: candidate_pool[node_id] *= 0.5 # 50% penalty
6
cross_encode_rerank — ms-marco-MiniLM-L-6-v2 + hybrid CE floor
Top 2×max_nodes candidates are jointly scored as (query, node_text) pairs by the cross-encoder. Unlike cosine similarity (which encodes query and node separately), the cross-encoder sees both together — capturing precise relevance signals bi-encoders miss.

Hybrid CE floor (FILCO-style, new in v3.1): always keep top-3, then keep extras within 4.0 of the best CE score. Pure ce_score > 0 floor drops everything when the model scores all candidates negative (common for abstract/architectural queries).

Model: 66 MB, pre-warmed on startup (~200ms disk load), CPU inference ~100ms for 15 candidates.
Hybrid floor logic (graph_rag.py)
max_ce = scored[0][0] # best CE score (already sorted desc) floor = max_ce - 4.0 kept = list(scored[:3]) # top-3 always kept extras = [item for item in scored[3:] if item[0] > floor] scored = kept + extras
7
_expand_full_bodies — read live source lines for top-5
After CE reranking, the top-5 survivors get their full_body field populated by reading actual lines from disk (line_start → line_end). This replaces the 300-char body_preview stored at index time — critical for large functions where the key logic is beyond the preview window. Failures silently fall back to body_preview. Only top-5 are expanded (disk I/O cost is proportional).
8
mmr_diversify — file-based diversity selection
Maximal Marginal Relevance selects the final max_nodes iteratively. Each candidate's effective score is reduced by 0.35 × count_already_selected_from_same_file. The third node from the same file needs a 0.70 score advantage over a fresh-file node to be selected. Prevents BFS clusters (all methods of one class) from dominating context.
MMR selection
adjusted = score - 0.35 × file_counts.get(file_key, 0) # Iteratively pick node with highest adjusted score # file_counts tracks how many nodes already selected from each file

The Four Specialist Agents

Explorer (Explain) — explorer.py + orchestrator.build_explain_context()
Path: Selection-aware → Graph RAG retrieval → SSE token streaming (bypasses LangGraph for explain to stream directly).
Selection injection: If user has a file/range selected in VS Code, the system finds the most specific graph node overlapping the midpoint of the selection range, injects it at the front of context, and rewrites the question to anchor to that symbol.
Virtual document guard: VS Code output panel URIs (extension-output-…) are detected and silently discarded.
Module-level fallback: If selected code is not a tracked graph symbol (e.g. top-level statements), raw lines are read from disk and injected as a synthetic CodeNode so the LLM still sees them.
Streaming: explore_stream() is an async generator — each LLM token becomes a SSE token event.
Debugger — debugger.py — 9-step graph-traversal algorithm
Step 1 — Entry discovery: Find nodes whose .name appears in the lowercased question. Fallback: node with highest in_degree (most-called function).
Step 2 — BFS forward: Follow CALLS edges up to max_hops=4 from each entry node.
Step 3 — 5-factor anomaly scoring per node:
FactorWeightComputation
Complexity0.30cyclomatic complexity / 10, clamped [0,1]
Error absence0.251.0 if no try/except/raise/catch in body; 0.0 otherwise
Keyword match0.20overlap(question_tokens, body_tokens) / len(question_tokens)
Out-degree coupling0.15out_degree / 10, clamped [0,1]
Inverted PageRank0.101 - min(pagerank × 5, 1.0) — low centrality = more suspect
Step 4: Rank top-5 by anomaly score. Step 5: Compute impact_radius (direct CALLS-predecessors of top suspect). Step 6: LLM diagnosis narrative constrained to traversal function names only (anti-hallucination).
Example anomaly_score
score = 0.30×complexity + 0.25×error_absence + 0.20×keyword_match + 0.15×coupling + 0.10×inverted_pagerank # e.g. login() — complexity=4, no error handling, "login" in question # → 0.30×0.4 + 0.25×1.0 + 0.20×1.0 + 0.15×0.3 + 0.10×0.8 = 0.795
Reviewer — reviewer.py — structured output + groundedness post-filter
Context assembly: target_node_id + its 1-hop CALLS predecessors (callers) + successors (callees). Each node labeled TARGET / CALLER / CALLEE in the prompt.
Range clause: if user has a line range selected, prompt gets "FOCUS: target your findings to lines X–Y of file.py".
Structured output: llm.with_structured_output(ReviewResult) returns Pydantic-validated findings. Each finding has: severity (critical/warning/info), category, description, file_path, line_start, line_end, suggestion.
Groundedness post-filter: any Finding whose file_path is not in the retrieved context set is dropped — Python check, not LLM judgment. Prevents hallucinated file paths.
GitHub PR integration: POST /review/post-pr sends findings as inline review comments via GitHub API (batched to 10, overflow → summary comment, 3× retry on 5xx).
Tester — tester.py — deterministic framework detection
Step 1 — Framework detection (no LLM): scans repo root for marker files. Priority: pytest → jest → vitest → junit → fallback: test_*.py exists → unknown.
Step 2 — Callee enumeration: CALLS-edge successors of target_node_id → mock targets list passed to prompt.
Step 3 — LLM structured output: Only test_code comes from LLM. test_file_path and framework are derived deterministically from (func_name, framework) — never trusted from LLM.
Step 4 — MCP file write: write_test_file() validates path (no "..", allowed extensions), creates parent dirs, writes to disk. Returns {success, path, error}. Extension shows "file written" badge vs "copy to clipboard" fallback.
Framework markers
pytest → pytest.ini, conftest.py, setup.cfg jest → jest.config.js / .ts / .json vitest → vitest.config.ts / .js junit → pom.xml, build.gradle # Fallback: any test_*.py in repo_root → pytest

The Critic — Deterministic Quality Gate

LLM-based critics add cost, non-determinism, and can be fooled by confident hallucinations. The Critic is purely rule-based — Python conditionals, zero API calls, microsecond execution.

Critic formula (critic.py)
score = 0.40 × groundedness + 0.35 × relevance + 0.25 × actionability # groundedness: |cited_nodes ∩ retrieved_set| / |cited_nodes| # relevance: structural quality (≥1 suspect + diagnosis / ≥1 finding + summary / def test_) # actionability: specificity (suspects/5 / suggestions ratio / test functions/3) # threshold: score < 0.7 → retry; loop_count ≥ 2 → force pass (max_critic_loops)

The Balanced Grounding Prompt — v3.1 Design

The system prompt in prompts.py went through two v3.1 iterations. An early version was over-restrictive ("ONLY use context") which caused poor answers on architectural questions where some general knowledge is needed. It was reverted. The current prompt is balanced: prefer context, but allow general programming reasoning to connect the dots.

prompts.py — SYSTEM_PROMPT (v3.1 balanced)
"""You are an expert code assistant. You answer questions about a codebase by reasoning from the retrieved code context provided. Rules: 1. Prefer the retrieved context as your primary source. When the answer is visible in the code blocks, base your answer on them and cite the exact file path and line range, e.g. `auth/login.py:42-55`. 2. Never fabricate a citation. Only cite file:line locations that appear verbatim in the context headers. 3. You may use your general programming knowledge to EXPLAIN or CONNECT what you see in the context (e.g. how a pattern works, what a base class implies). Do NOT state facts about this specific codebase that contradict or go beyond the retrieved context. 4. If the retrieved context does not contain enough information to answer confidently, say so briefly and share what you can infer from what was retrieved. 5. Do not invent function names, class names, or file paths not present in context. 6. Keep answers concise and focused on what the code actually shows.""" # Key design decision (v3.1): # Rule 3 distinguishes "explain a pattern you see" (allowed general knowledge) # vs "assert a codebase fact" (requires context support). # The over-restrictive version blocked useful answers like explaining what # a design pattern implies — reverted in 8d36aab, replaced by this.
Over-Restrictive (reverted)
  • Answer ONLY from the context block
  • Refused architectural questions needing general knowledge
  • "I'm not certain" on every question about patterns
  • Reverted in commit 8d36aab
Balanced Grounding (v3.1 current)
  • PREFER context, CITE from context
  • General knowledge allowed to explain/connect patterns
  • Can say "this looks like a factory pattern" even if not in docstring
  • Still prohibits fabricating file:line citations or codebase facts

Anti-Hallucination Layers

1. Retrieval Constraint
LLM only sees the retrieved context block. Can't fabricate what it wasn't shown.
2. Balanced Prompt (v3.1)
"Never fabricate a citation." General knowledge allowed to explain patterns in context, not to assert codebase facts.
3. Groundedness Post-Filter
Reviewer drops any Finding whose file_path is not in the retrieved node set — Python check, not LLM.
4. Traversal Grounding
Debugger prompt says "only mention function names from this list: {traversal_names}". PPR/BFS produces the list, not the LLM.
5. Critic Hard Cap
max 2 retries (loop_count ≥ 2 → force passed=True). LangGraph loop always terminates. Best answer always returned.
6. Deterministic Paths
test_file_path and framework come from Python logic (marker file scan, string formatting) — never from LLM output.

Side-Effect Layer

MCP Tool Layer

The MCP layer (app/mcp/tools.py) provides the two side-effect I/O operations that agents can trigger. Neither function imports from app.agent.* at module level (no circular imports). Both are intentionally stateless — safe to call from any request.

post_review_comments()
Posts Reviewer findings as GitHub PR inline review comments via GitHub REST API.

Batching: Up to 10 findings → single POST /pulls/{pr}/reviews batch. Findings 11+ → one summary issue comment.
422 fallback: If batch review returns 422 (invalid line position in at least one comment), retries each finding individually. Skips findings where line position is invalid.
Retry policy: tenacity — 3 attempts, exponential backoff [1, 8]s, retry on 5xx only (422 is permanent, never retried).
Token security: GITHUB_TOKEN stored server-side in .env (Settings). Extension sends findings + PR metadata; token never exposed to frontend.
POST /review/post-pr
{ "findings": [/* list of Finding dicts */], "repo": "owner/repo", "pr_number": 42, "commit_sha": "abc123" } → {"posted": 8, "skipped": 1, "summary_posted": false}
write_test_file()
Writes generated test code to disk from TestResult. Security-validated before any file operations.

Validation order:
1. Reject ".." in raw path string (path traversal — checked BEFORE Path.resolve())
2. Reject disallowed extension (allowlist: .py .ts .js .tsx .jsx .java .go)
3. Reject existing file when overwrite=False
4. Create parent directories (mkdir -p)
5. Write file

Why check ".." before Path.resolve()? resolve() normalizes paths — tests/../../../etc/cron.d/attack looks relative but resolves outside repo. Raw string check prevents normalization from hiding the traversal.
write_test_file return
# Success: {"success": True, "path": "/repo/tests/test_login.py", "error": None} # Traversal rejected: {"success": False, "path": None, "error": "path traversal rejected: '..' found in path"}

Architecture Trade-offs

Key Design Decisions

Every architectural choice has a "why" and a "what you give up." These are the questions interviewers probe.

SQLite vs. PostgreSQL + pgvector
✓ Chose: SQLite + sqlite-vec
✗ Rejected: PostgreSQL
Zero infrastructure. Works on install. Each workspace gets its own isolated file. The tradeoff: no horizontal scaling — fine for single-user dev tool.
Sidecar process vs. in-extension Python
✓ Chose: Persistent FastAPI sidecar
✗ Rejected: Per-query subprocess
Graph loads once, reused for all queries. Per-query subprocess would reload the NetworkX graph every time (500ms–2s overhead on large repos).
Graph RAG vs. plain vector search
✓ Chose: Semantic + FTS5 + BFS
✗ Rejected: Cosine-only
Context Precision: 0.3706 (v2+CE) vs 0.1585 (naive) — +134%. The bug is often one CALLS hop away from the function you asked about; cross-encoder reranking further tightens the context window.
Deterministic Critic vs. LLM Critic
✓ Chose: Rule-based Python checks
✗ Rejected: LLM self-evaluation
Zero API cost, reproducible scores, not fooled by confident-sounding hallucinations. Rule checks (non-empty suspects, ≥1 def test_, etc.) capture quality for structured outputs.
tree-sitter vs. LSP for call graph
✓ Chose: tree-sitter
✗ Rejected: Language Server Protocol
tree-sitter is faster (pure parsing, no server round-trips), language-server-independent, and works offline. LSP gives more accurate dynamic dispatch — tradeoff accepted.
Lazy imports in agent functions
✓ Chose: Import inside function body
✗ Rejected: Module-level imports
During pytest collection, module-level imports trigger get_settings() before fixtures run → ValidationError (no API keys). Lazy imports prevent this.
Honest Assessment

Known Weaknesses & Gaps

Understanding limitations is as important as knowing strengths. These come up in senior interviews.

⚠
Static analysis only. tree-sitter extracts CALLS from source text. Dynamic dispatch (Python duck typing, JS callbacks, virtual method overrides) produces no edges. The call graph is an approximation — polymorphic calls are invisible.
⚠
Truncated body previews. CodeNode.body_preview stores only the first N characters of a function. Critical logic in large functions may be in the truncated part — the LLM never sees it.
⚠
Single-process retrieval. Vector search + BFS run in the FastAPI process handling all requests. On large repos (50K+ nodes), BFS at depth 2 can block other requests for several hundred milliseconds. A worker queue would fix this.
💡
If rewriting retrieval: (1) Replace cosine similarity with a learned re-ranker — done: cross-encoder (ms-marco-MiniLM-L-6-v2) now reranks top candidates. (2) Add backward CALLS expansion (who calls this? — current BFS is forward-only). (3) Fetch actual source lines at query time instead of stored body_preview — larger context, fewer blind spots in large functions.

Where do I start to understand the full data flow?
Start with the Architecture Overview above for the big picture. Then follow the data in order: Ingestion (walker → AST → graph → embedder) → Retrieval (graph_rag.py) → Agents (router → specialist → critic) → API (SSE events) → Extension (sidebar → webview). For a 30-minute orientation: sections 00, 07, 08, 21 in docs/guide.
How do I add a new LLM provider (e.g. Cohere)?
5 changes: (1) model_factory.py — add elif in get_llm() and get_embeddings(). (2) Install LangChain integration package. (3) config.py — add API key field to Settings. (4) App.tsx — add to intent selector dropdown. (5) ConfigManager.ts — add key to requiredKeys map.
How do I add a new specialist agent (e.g. Refactor)?
(1) Create refactorer.py with RefactorResult Pydantic model + refactor() function (lazy-import pattern). (2) Register agent node in orchestrator.py (LangGraph state machine). (3) Add "refactor" to Router's intent options in router.py. (4) Add "refactor" to intent selector in App.tsx. (5) Handle result SSE event for intent == "refactor" in SidebarProvider.ts.
How do I debug the Python backend locally?
cd backend && python -m uvicorn app.main:app --reload --port 8000. Auto-generated docs at http://localhost:8000/docs. API surface: POST /index, GET /index/status, DELETE /index, POST /query (SSE), POST /api/config, GET /health.
Backend isn't starting — where do I look?
(1) VS Code Output panel → "Nexus" — SidecarManager logs stdout/stderr. (2) Check for Pydantic ValidationError (missing env var), ImportError (missing dep), or port conflicts. (3) Check .nexus/sidecar.lock — stale PID from crashed process? Delete it to force fresh start. (4) GET /health → 200 means backend is up; check for port mismatch.
What is .nexus/ and should I gitignore it?
Yes, always gitignore it. Contains graph.db (SQLite index, 5–50 MB) and sidecar.lock (PID/port). Machine-specific absolute paths, local embeddings. Committing it causes conflicts across team members.
How do I run the RAGAS evaluation after changing retrieval?
python eval/run_ragas_redesign.py --limit 30 --workers 1 for a full 30Q run (both v2 and v2+CE pipelines). Use --ce-only to skip re-running v2 and load the prior result instead (~1hr vs ~2hrs). Results land in eval/results/ragas_redesign_*.json. Key metrics: faithfulness, answer_relevancy, context_precision.
Why Graph RAG over plain vector similarity search?
Plain vector search finds semantically similar code, not causally related code. "Why does login fail?" returns functions whose names look like login. Graph-RAG starts from those seeds then uses Personalized PageRank over CALLS+CLASS_CONTAINS edges to discover multi-hop neighbors — what login calls, what calls login, and methods of the same class — all proportionally weighted by how strong the original seed was. Then a cross-encoder (ms-marco-MiniLM-L-6-v2, 66MB) jointly scores (query, node_text) pairs for precise relevance discrimination. Results: faithfulness=0.54, answer_relevancy=0.48, context_precision=0.37 on golden_qa_v2 30Q. +46% precision vs v2 alone (BFS), +134% vs naive vector (0.1585).
Why NetworkX in-memory rather than Neo4j?
NetworkX runs traversals (BFS, PageRank, degree centrality) as direct library calls inside the Python process — no query language, no network round-trip. Neo4j would add serialization overhead + operational complexity for what is a local-first tool. Graph persists to SQLite, loaded into NetworkX on startup.
How do you ensure the LLM doesn't hallucinate citations?
Four layers: (1) Retrieval constraint — LLM only sees the retrieved context block. (2) Prompt constraint — "Never fabricate a citation." (3) Post-filter — Reviewer drops findings with file_path not in retrieved set (Python check). (4) Critic groundedness — traversal_path is set by BFS code (deterministic), not the LLM; suspects outside it penalize the score.
Why is the Critic rule-based instead of LLM-based?
LLM critics: extra API cost, non-deterministic (same output → different score), fooled by confident-sounding hallucinations. Rule-based: zero cost, deterministic, fast (Python conditionals in microseconds). For structured outputs (non-empty suspects, ≥1 def test_, every finding has a suggestion), rule checks capture quality well.
Walk me through latency from Enter to first token.
React postMessage → SidebarProvider captures active editor file+range → SseStream opens fetch to POST /query → FastAPI validates index status → loads nx.DiGraph from cache → embeds query → cosine ANN with min_similarity=0.15 floor → FTS5 BM25 stopword-filtered search → RRF merge (k=60) → Personalized PageRank expansion over CALLS+CLASS_CONTAINS edges → combine + test-file penalty → cross-encoder rerank ms-marco-MiniLM (pre-warmed, ~100ms CPU) → hybrid CE floor → full-body expansion for top-5 → MMR diversity selection → Explorer streams first LLM token via SSE → SseStream parses token event → webview.postMessage → React renders. Typical: 300–800ms to first token on warm graph (CE dominates).
How does the sidecar handle multiple VS Code windows?
Lockfile singleton: .nexus/sidecar.lock = PID + port. Second window opening same repo reads lockfile and reuses existing sidecar. Different repo → new sidecar on different port. Sidecar self-terminates after 600s idle (_IDLE_TIMEOUT in main.py).
How does CI ensure the shipped binary actually works?
After each PyInstaller build: (1) Launch binary with --port 0 (OS-assigned). (2) Read port from stdout. (3) POST /index small test repo. (4) Wait for GET /index/status → "complete". (5) POST /query test question. (6) Assert response has done SSE event. Runs on macOS + Windows + Linux matrix.
Why lazy imports inside agent function bodies?
During pytest collection, Python imports every test file. Module-level get_settings() / get_llm() would run before test fixtures — triggering Pydantic ValidationError (no API keys). Importing inside the function body means the import only runs when the function is called. Tests can import debugger.py to test _score_node without live API keys.
Does it work on my repo right now, or does it need special setup?
Works on any repo once indexed. Click "Index Workspace" in sidebar. Nexus scans every .py/.ts/.js/.tsx/.jsx/.java/.go/.rs file, builds call graph, stores embeddings locally. ~5,000-function repo takes 30–90 seconds. You need an API key for at least one provider (Ollama is the zero-cost local option).
Will it leak my code to the internet?
Only through the LLM API you configure — same data privacy as using those APIs directly. The full index never leaves your machine. Use Ollama for zero-network operation. There is no Nexus server — the backend runs as a local process on your machine.
What happens when I edit a file? Does it auto-reindex?
Yes. FileWatcher.ts watches for file changes. 2-second debounce after saves. Only changed files sent for incremental reindex. Backend re-parses, removes old nodes, inserts new ones. Typically completes in <5 seconds per file.
How is Nexus different from asking ChatGPT about my code?
Three differences: (1) Scope — Nexus knows your whole codebase (every function, every call relationship). ChatGPT only knows what you paste. (2) Citations — Nexus cites specific file:line locations, clickable to jump directly. (3) Freshness — Nexus indexes your latest code on every save.
Can it work with a monorepo with multiple languages?
Yes. Walker detects language from file extension; ast_parser dispatches to the appropriate tree-sitter grammar. Python + TypeScript repos get a mixed graph. CALLS edges across language boundaries aren't extracted (Python→TypeScript calls), but each language's internal call graph is complete.
How does Debug mode differ from searching for the error message?
Search finds where a string appears. Debug mode finds why. The Debugger scores functions on 5 factors: complexity, missing error handling, keyword relevance, coupling, low centrality. Returns top 5 suspects with anomaly scores and a diagnosis narrative. Clicking a suspect opens the file at that line.
What's the biggest weakness of the current architecture?
Three honest weaknesses: (1) Static analysis only — dynamic dispatch (duck typing, callbacks, virtual methods) produces no CALLS edges. (2) Truncated body previews — critical logic in large functions may be cut. (3) Single-process retrieval — BFS at depth 2 on large repos can block other requests. A worker queue would fix the third.
Why does write_test_file check for ".." before Path.resolve()?
Path.resolve() normalizes paths and follows symlinks. An attacker could craft tests/../../../etc/cron.d/attack — it looks like a relative path, but after resolve() points outside the repo. Checking the raw string for ".." before any Path operation prevents traversal normalization from obscuring the attack. Only after the raw check passes is Path construction safe.
How would you scale this to work across a team?
Current arch: strictly single-user local. To scale: (1) Move graph.db to shared store (shared FS or PostgreSQL + pgvector). (2) Add auth to backend API (currently no auth). (3) Replace sidecar with persistent service (container/pod). (4) The repo_path scoping in the graph schema already supports multi-repo namespacing — no schema change needed. Agent + retrieval layers unchanged.
If you rewrote retrieval from scratch, what would change?
(1) BFS expansion — shipped: replaced by Personalized PageRank which is multi-hop, score-proportional, and CLASS_CONTAINS-aware. (2) Pure CE score floor — shipped: hybrid CE floor keeps top-3 always, then within 4.0 of best score (handles architectural queries where all CE scores are negative). (3) body_preview truncation — shipped: full-body expansion reads live source lines for top-5 CE survivors. (4) Next: semantic filtering of CALLS neighbors (select by cosine sim to query, not just pagerank) to avoid high-degree utility nodes. (5) Next: multi-provider embedding fusion — combine Mistral+OpenAI embeddings, re-rank by agreement.
The Reviewer uses with_structured_output — what happens under the hood?
LangChain inspects the Pydantic model's JSON schema. For OpenAI/Mistral with tool-calling: schema becomes a tool definition; LLM is asked to call it. For providers without tool-calling: JSON mode with schema embedded in system prompt. LangChain deserializes response into ReviewResult via Pydantic. If JSON is malformed: ValidationError → Critic score 0.0 → retry.
