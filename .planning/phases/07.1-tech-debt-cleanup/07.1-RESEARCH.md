## Summary

Two non-critical gaps were identified in the v1.0 audit. Both are small, targeted changes with no architectural impact and no new dependencies.

**Gap 1 — Stale FTS5 rows on incremental re-index.** When `run_ingestion()` is called with `changed_files`, the pipeline calls `delete_nodes_for_files(changed_files, repo_path)` which correctly cleans `graph_nodes` and `graph_edges` in SQLite. However it does NOT clean `code_fts` (the FTS5 virtual table in the same SQLite file). The embedder has `delete_embeddings_for_repo()` for full-repo deletion but has no per-file equivalent. As a result, if a function is renamed or a file is deleted, its old FTS5 row persists and will be returned by `name MATCH` queries forever. The fix is: add `delete_embeddings_for_files(file_paths, repo_path)` to `embedder.py` and call it from `run_ingestion()` in the incremental path alongside the existing `delete_nodes_for_files()` call.

**Gap 2 — Missing backend Docker healthcheck.** The `backend` service in `docker-compose.yml` has no `healthcheck` stanza, unlike `postgres` which has a full healthcheck with `pg_isready`. The `GET /health` endpoint already exists in `main.py` (returns `{"status": "ok", "version": "1.0.0"}`). The fix is a 5-line `healthcheck` block in `docker-compose.yml` using `curl -f http://localhost:8000/health`. No code changes are needed in the backend itself.

**Primary recommendation:** Implement `delete_embeddings_for_files(file_paths, repo_path)` in `embedder.py` mirroring the pattern of `delete_embeddings_for_repo()`, wire it into the `run_ingestion()` incremental path, then add the `healthcheck` stanza to the backend service in `docker-compose.yml`.

### Pattern 1: File-scoped FTS5 Deletion

**What:** Delete FTS5 rows by `file_path` column before re-ingesting changed files.
**When to use:** Incremental re-index (any call with `changed_files` list).
**How it works:** The `code_fts` table stores `node_id`, `name`, and `file_path`. Since `file_path` is UNINDEXED (stored but not tokenized), queries against it use `WHERE file_path = ?` not `MATCH`. The delete is straightforward:

```python
# Source: existing embedder.py pattern (delete_embeddings_for_repo uses same conn style)
def delete_embeddings_for_files(file_paths: list[str], repo_path: str) -> None:
    """Delete pgvector rows and FTS5 rows for the given file_paths within a repo.

    Called before re-parsing changed files in the incremental re-index path.
    Mirrors delete_nodes_for_files() in graph_store.py.

    Args:
        file_paths: List of file_path values matching the file_path column.
        repo_path:  Repository identifier — scopes the pgvector delete.
    """
    if not file_paths:
        return

    # --- pgvector: delete by (repo_path, file_path) ---
    pg_conn = get_db_connection()
    register_vector(pg_conn)
    try:
        placeholders = ", ".join(["%s"] * len(file_paths))
        with pg_conn.cursor() as cur:
            cur.execute(
                f"DELETE FROM code_embeddings "
                f"WHERE repo_path = %s AND file_path IN ({placeholders})",
                [repo_path, *file_paths],
            )
    finally:
        pg_conn.close()

    # --- FTS5: delete by file_path column (UNINDEXED = stored, use WHERE not MATCH) ---
    db_path = _sqlite_db_path()
    sqlite_conn = sqlite3.connect(db_path)
    try:
        placeholders = ", ".join(["?"] * len(file_paths))
        sqlite_conn.execute(
            f"DELETE FROM code_fts WHERE file_path IN ({placeholders})",
            file_paths,
        )
        sqlite_conn.commit()
    finally:
        sqlite_conn.close()
```

**Critical FTS5 detail:** `file_path` is declared `UNINDEXED` in the `code_fts` CREATE statement. This means it is stored in the FTS5 table but NOT tokenized/indexed for MATCH queries. To delete rows by it, use a plain `WHERE file_path IN (...)` clause — do NOT use the FTS5 `MATCH` syntax. This is the correct, working approach.

The `python:3.11-slim` base image includes `curl` by default. The `/health` endpoint is already live at startup (FastAPI registers routes before lifespan completes the `yield`).


**start_period rationale:** 15s gives uvicorn + DB init time to complete. The postgres service uses `start_period: 10s`; backend needs slightly more since it depends on postgres being ready first.

### Anti-Patterns to Avoid

- **Using FTS5 MATCH to delete:** `DELETE FROM code_fts WHERE name MATCH 'func_0'` — MATCH is for full-text search, not exact equality. Use `WHERE file_path = ?` for non-tokenized columns.
- **Deleting only from FTS5 but not pgvector:** Both stores hold redundant data scoped by `file_path`. Delete from both in `delete_embeddings_for_files()`.
- **Using `wget` instead of `curl` in healthcheck:** `python:3.11-slim` has curl; using CMD-SHELL with wget requires verification of its presence. Stick with curl.
- **Using `CMD-SHELL` format when `CMD` format works:** `["CMD", "curl", "-f", "url"]` is preferred over `["CMD-SHELL", "curl -f url"]` — avoids shell injection, cleaner.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| FTS5 row existence check before delete | Custom SELECT then DELETE logic | Single DELETE WHERE — no-op if row absent | SQLite DELETE on nonexistent rows is a no-op, no need to check first |
| pgvector file-scoped delete | Iterating node IDs then deleting | Single DELETE WHERE file_path IN (...) | pgvector table has `file_path` column — direct clause is O(1) with index |

## Common Pitfalls

### Pitfall 1: FTS5 UNINDEXED Column Queries
**What goes wrong:** Attempting `WHERE name MATCH 'file_path_value'` or using FTS5 special syntax on an UNINDEXED column.
**Why it happens:** Developers familiar with FTS5 MATCH syntax try to apply it to non-tokenized columns.
**How to avoid:** UNINDEXED columns are plain stored columns — use standard SQL `WHERE file_path = ?` or `WHERE file_path IN (...)`.
**Warning signs:** `OperationalError: no such column: file_path` in FTS5 context — means you're mixing FTS5 special syntax with a regular column reference.

### Pitfall 2: FTS5 Integrity After Delete
**What goes wrong:** FTS5 virtual tables maintain internal data structures. After deleting rows, the internal B-tree can have ghost entries if the table was created with `content=''` (contentless mode).
**Why it happens:** The current `code_fts` does NOT use contentless mode (confirmed: `content=''` was removed in Phase 05, noted in STATE.md decisions). So standard DELETE works correctly.
**How to avoid:** No special handling needed. The current schema is `CREATE VIRTUAL TABLE code_fts USING fts5(node_id UNINDEXED, name, file_path UNINDEXED)` — a regular FTS5 table, not contentless.
**Warning signs:** If `content=''` were present, DELETE would appear to succeed but SELECT would still return deleted rows.

### Pitfall 3: Docker Healthcheck start_period Too Short
**What goes wrong:** The healthcheck starts probing before uvicorn has bound the port, causing false unhealthy status and downstream `depends_on` failures.
**Why it happens:** FastAPI + uvicorn can take 2-8s to start on a fresh container, especially if DB init (pgvector extension creation) blocks.
**How to avoid:** Use `start_period: 15s`. Failures during start_period do not count toward `retries`.
**Warning signs:** Backend container flips to `unhealthy` immediately after starting even though the app works fine afterward.

### Pitfall 4: Missing Import in pipeline.py
**What goes wrong:** Adding the call to `delete_embeddings_for_files()` but forgetting to add it to the import line.
**Why it happens:** The existing import from embedder only imports `embed_and_store`.
**How to avoid:** Update line 8 of `pipeline.py`:
```python
# Before:
from app.ingestion.embedder import embed_and_store
# After:
from app.ingestion.embedder import embed_and_store, delete_embeddings_for_files
```
