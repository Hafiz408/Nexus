**`backend/app/models/schemas.py`** — define two Pydantic v2 models:

```python
from pydantic import BaseModel

class CodeNode(BaseModel):
    node_id: str          # "relative_file_path::name" format (PARSE-04)
    name: str             # function/class/method name
    type: str             # "function", "method", "class"
    file_path: str        # absolute path to source file
    line_start: int       # 1-indexed start line
    line_end: int         # 1-indexed end line
    signature: str        # declaration line(s) before body
    docstring: str | None = None   # extracted docstring (PARSE-05)
    body_preview: str = ""         # first 300 chars of body (PARSE-05)
    complexity: int = 1            # keyword count proxy, min 1 (PARSE-05)
    embedding_text: str = ""       # "{signature}\n{docstring}\n{body_preview}" (PARSE-06)

class CodeEdge(BaseModel):
    source_id: str        # node_id of the source node
    target_name: str      # unresolved name (module or function name)
    edge_type: str        # "IMPORTS" or "CALLS"
```

Use `str | None` syntax (Python 3.10+ union, compatible with Python 3.11 project target — NOT `Optional[str]`).

Do NOT use `Optional` from typing — the project already uses pydantic v2; `str | None` is idiomatic.

These exact versions are required for API compatibility (as documented in research):
- tree-sitter 0.25.2: `Parser(Language(...))` constructor API, `captures()` returns dict
- tree-sitter-python 0.25.0: `tspython.language()` returns capsule object (not int)
- tree-sitter-typescript 0.23.2: exposes `language_typescript()` and `language_tsx()` (NOT `.language()`)

Do NOT use `>=` version constraints — the API changed significantly at 0.21 and pinning prevents breaking changes from a pip update.
