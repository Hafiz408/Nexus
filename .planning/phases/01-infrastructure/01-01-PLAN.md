<task type="auto">
  <name>Task 1: Create docker-compose.yml with postgres health check and backend service</name>
  <files>docker-compose.yml</files>
  <action>
Create `docker-compose.yml` at the repo root with two services:

**postgres service:**
- Image: `pgvector/pgvector:pg16` (NOT `postgres:16` — this image has pgvector pre-installed, no custom init needed)
- Container name: `nexus_postgres`
- Environment variables sourced from `.env` via `${VAR}` syntax: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`
- Ports: `5433:5432` (map to host port 5433 to avoid conflicts with any local postgres already on 5432; document this in a comment)
- Volumes: named volume `postgres_data:/var/lib/postgresql/data`
- Healthcheck: `pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}` with interval 5s, timeout 5s, retries 10, start_period 10s

**backend service:**
- Build context: `./backend`
- Container name: `nexus_backend`
- `depends_on: postgres: condition: service_healthy` — Docker Compose v2 syntax; this is MANDATORY to prevent backend starting before postgres accepts connections
- `env_file: .env` — loads all environment variables from .env file
- Volumes: `./data:/app/data` (bind mount, NOT a named volume — INFRA-04 requires SQLite files to persist on host and be visible in the project directory)
- Ports: `8000:8000`

**Volumes section:** Declare `postgres_data:` as a named volume at the top-level.

**Anti-patterns to avoid:**
- Do NOT hardcode any secret values (passwords, keys) directly in the compose file
- Do NOT use `depends_on` without `condition: service_healthy` — bare `depends_on` only waits for container start, not postgres readiness
- Do NOT use `pgvector/pgvector:latest` — pin to `:pg16` explicitly
- Do NOT use `docker-compose` (hyphenated v1 syntax) — this project uses Docker Compose v2 (`docker compose`)
  </action>

<task type="auto">
  <name>Task 2: Create backend/Dockerfile and backend/requirements.txt</name>
  <files>backend/Dockerfile, backend/requirements.txt</files>
  <action>
Create `backend/` directory if it does not exist.

**backend/Dockerfile:**
```
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--reload"]
```

Key points:
- `python:3.11-slim` NOT `python:3.11-alpine` — Alpine causes build failures with psycopg2-binary and numpy due to missing C headers
- Copy `requirements.txt` BEFORE `COPY . .` — Docker layer cache skips `pip install` on code-only changes
- `--reload` is appropriate for local dev; remove in production

**backend/requirements.txt** — include all Phase 1 deps plus stubs needed by later phases (so the Dockerfile builds a complete dev environment):
```
fastapi>=0.115.0
uvicorn[standard]
pydantic-settings>=2.0.0
psycopg2-binary
pgvector
numpy
aiosqlite
```

Use `psycopg2-binary` (NOT bare `psycopg2`) — the slim base image lacks `libpq-dev` and `gcc`, so the source build would fail. The binary package bundles the compiled C extension.

Also create a minimal `backend/app/__init__.py` and `backend/app/main.py` so uvicorn has a valid entry point to start (just `from fastapi import FastAPI; app = FastAPI()` — the full implementation is in later phases):

`backend/app/__init__.py`: empty file
`backend/app/main.py`:
```python
from fastapi import FastAPI

app = FastAPI(title="Nexus API", version="1.0.0")

@app.get("/health")
def health():
    return {"status": "ok", "version": "1.0.0"}
```
